## 工作流编排

### 1. 计划节点默认
- 任何非 trivial 任务（3+ 步骤或架构决策）都进入计划模式
- 如果出现问题，立即停止并重新规划——不要继续硬推
- 将计划模式用于验证步骤，而不仅是构建
- 预先编写详细规范以减少歧义

### 2. 子代理策略
- 自由使用子代理以保持主上下文窗口干净
- 将研究、探索和并行分析卸载给子代理
- 对于复杂问题，通过子代理投入更多计算资源
- 每个子代理专注于一个任务

### 3. 自我改进循环
- 用户任何纠正后：将模式更新到 `tasks/lessons.md`
- 为自己编写规则以防止同样错误
- 无情地迭代这些教训直到错误率下降
- 会话开始时回顾相关项目的教训

### 4. 完成前验证
- 从未证明有效前不要标记任务完成
- 在相关时对比 main 和你的更改之间的行为差异
- 问自己："staff engineer 会认可这个吗？"
- 运行测试、检查日志、证明正确性

### 5. 追求优雅（平衡）
- 对于非 trivial 更改：暂停并问"有更优雅的方法吗？"
- 如果修复感觉 hacky：" knowing everything I know now, implement the elegant solution"
- 简单、明显的修复跳过这个——不要过度工程
- 呈现前先挑战自己的工作

### 6. 自主 Bug 修复
- 给出 bug 报告时：直接修复。不要寻求手把手指导
- 指向日志、错误、失败测试——然后解决它们
- 用户零上下文切换需求
- 无需被告知如何处理就去修复失败的 CI 测试

## 任务管理
1. **先计划**：写入计划到 `tasks/todo.md`，包含可勾选项目
2. **验证计划**：开始前确认
3. **跟踪进度**：完成时标记项目
4. **解释更改**：每步提供高层摘要
5. **记录结果**：添加审查部分到 `tasks/todo.md`
6. **捕获教训**：纠正后更新 `tasks/lessons.md`

## 核心原则
- **简洁优先**：让每个更改尽可能简单。影响最小代码。
- **不懒惰**：找到根本原因。无临时修复。高级开发者标准。
- **最小化影响**：更改应只触及必要内容。避免引入 bug。
