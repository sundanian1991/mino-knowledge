提出者：Matt Pocock（TypeScript社区核心影响力人物，Total TypeScript课程创始人）
核心理念：让AI选择做什么，不是你告诉它做什么
技术栈：Claude Code + Skill + Ralph脚本 + Docker沙盒
工作流特点：将80%时间投入需求工程，实现执行环节的自动化循环，用户仅需最终人工验收
📋 四步核心工作流详解
🔥 第一步：/write-a-prd - 模糊想法→精确需求
核心功能：将抽象概念转化为可执行的产品需求文档（PRD）
PRD Skill执行流程：
引导用户详细描述问题与想法
自动检索代码库验证需求可行性
无情访谈：通过多轮提问消除需求歧义，解决依赖关系
识别"深模块"机会（小接口、大实现，源自《软件设计哲学》）
确定测试范围与策略
生成GitHub Issue格式的PRD文档
PRD包含要素：问题陈述、解决方案、用户故事（要求"极其详尽"）、实现决策、测试决策、范围外事项
🔥 第二步：/prd-to-issues - PRD→垂直切片看板
核心功能：将PRD拆解为独立可执行的任务单元
切片方式
定义
优缺点
水平切片
按技术层拆分（先数据库→再API→再UI）
❌ 周期长，集成风险高
垂直切片
每个任务贯穿全技术栈（端到端实现）
✅ 可独立验证，风险分散
切片要求：
每个切片可独立演示验证
首个切片为"hello world"级最简实现
后续切片逐步扩展边界情况与用户故事
任务管理：按依赖顺序创建GitHub Issues，形成可视化看板，并确认粒度、依赖关系、优先级与完整性
🔥 第三步：ralph.sh - 自动循环写代码
核心功能：30行bash脚本实现AI自动化编码循环（Docker沙盒环境运行）

评估代码质量与架构合理性
定位：工作流中唯一需要人工参与的环节

💡 设计哲学与关键洞察
1. 深模块优先
源自John Ousterhout《A Philosophy of Software Design》，强调小接口隐藏复杂实现
2. 垂直切片方法论
端到端任务拆分，确保每个交付单元可独立验证，降低集成风险
3. 访谈驱动开发
通过"无情访谈"消除需求歧义，解决80%的AI编程返工问题
4. GitHub Issue为中心
将AI定位为规划工具而非执行工具，通过Issues实现任务可视化
5. 反模式明确化
每个Skill均定义禁止行为，比正向指导更能保证质量
📝 补充细节
Ralph命名由来：源自《辛普森一家》角色Ralph Wiggum，寓意"看似简单却偶现天才操作"
CLI vs GUI：选择Claude Code CLI而非Cursor的原因在于支持脚本化、循环执行与无人值守
时间分配原则：80%时间用于PRD与任务拆分，20%时间用于自动化执行与验收
