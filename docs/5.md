🔑 六大核心经验解析
一、多模型对审：Plan阶段填坑
核心方法：
Opus4.6（Plan Mode）：生成方案文档
Codex5.3 Extra High：评审方案，进行5-6轮质疑补全
价值：将修改循环前移至纯文本阶段，执行阶段几乎一次通过
关键发现：
写代码成本远高于审方案成本，早期方案打磨可降低70%返工率
单一模型易陷入思维盲区，多模型交叉验证可提升方案健壮性
二、分形文档+强制同构
三层文档架构（受@赵纯想启发）：
层级
内容
作用
L1（根目录）
全局地图（技术栈/目录结构/架构决策）
提供项目全景视图
L2（模块目录）
成员清单+接口说明
明确模块边界与依赖
L3（文件头部）
依赖声明+职责描述
单个文件功能定位
核心机制：
CLAUDE.md强制加载：确保AI随时获取完整上下文
回环检查：代码变更后自动同步文档（依赖声明/成员清单），避免文档代码脱节
成效：Opus4.6自评"文档同构做得最好的前端项目之一"，L2模块覆盖率达94.7%（18/19）。
三、Hook自动拦截：规范执行保障
技术实现：
将项目规范编码为脚本，通过Git Hook拦截违规代码
案例：强制使用Arco Design组件，实现"零逃逸"
优势对比：
方式
效果
维护成本
Prompt约束
执行不稳定，易遗忘
高（需反复强调）
Hook拦截
100%执行规范
低（一次配置永久生效）
四、Plan文件持久化：状态可靠管理
关键措施：
修改plan Directory配置，使Plan文档存储于项目目录内
新增Checklist跟踪进度，会话中断后可从文件断点恢复
价值：文件系统作为唯一可靠状态源，解决AI对话记忆有限问题，支持多设备协同开发。
五、质量红线量化：硬阈值管控
核心指标（硬阈值）：
单文件 ≤ 800行
单函数 ≤ 30行
嵌套层级 ≤ 3层
分支数量 ≤ 3个
原理：AI对"短函数"等模糊指令理解不稳定，但对具体数字阈值执行精确，可将代码质量问题降低60%。
六、渐进式上下文加载：信息效率优化
策略：
改Bug：仅加载目标模块文档
架构决策：加载全局地图
类比：如数据库索引设计，优先保证信息命中率
必要性：即使Opus4.6的1M上下文，仍需通过渐进加载控制熵增，避免信息过载。
🎯 核心洞察
对抗上下文熵增：所有机制共同目标是解决AI协作中的信息衰减与混乱问题，迭代速度越快，该问题越关键
文档即系统：Opus4.6对分形文档评分85分，证明"好文档是好系统的底子"
工程化大于Prompt：通过Hook、分形文档等工程手段，比单纯优化Prompt更可靠
